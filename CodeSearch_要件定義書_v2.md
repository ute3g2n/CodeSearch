# CodeSearch — 要件定義書

**プロジェクト名:** CodeSearch
**文書バージョン:** 2.4
**作成日:** 2026-03-26
**ステータス:** レビュー待ち（複数インスタンス起動仕様追加）

---

## 1. プロジェクト概要

### 1.1 目的

VS Codeの検索機能とコードビュー機能を抽出・特化させた、高速全文検索デスクトップアプリケーション。VS Codeにはないインデックスベースの全文検索により、中〜大規模プロジェクトにおいても高速な検索体験を提供する。

### 1.2 コンセプト

「VS Code − 編集機能 + 全文検索インデックス + カラーブックマーク」

### 1.3 ターゲットユーザー

ソースコードやログファイル等の大量テキストファイルを横断検索する必要があるソフトウェア開発者・インフラエンジニア・テクニカルライター。

## 2. 技術スタック

| 領域 | 技術 | 選定理由 |
|------|------|----------|
| アプリフレームワーク | **Tauri v2** (Rust + WebView2) | 軽量（数MB）、Rustバックエンドと自然に統合、Windows WebView2同梱不要（Win11標準搭載） |
| フロントエンド | **React + TypeScript + Vite** | 高い生産性、豊富なエコシステム |
| 全文検索エンジン | **Tantivy** (Rust) | Rust製で高速、日本語トークナイザ（lindera）対応、Tauriバックエンドと同一プロセスで動作 |
| シンタックスハイライト | **TextMate文法** (`.tmLanguage` / `.tmLanguage.json`) | VS Code互換の網羅性、既存文法定義ファイルを流用可能 |
| エンコーディング判定 | **encoding_rs** + **chardetng** (Rust) | Shift-JIS/EUC-JP/UTF-8等の自動判定 |
| ファイル監視 | **notify** (Rust) | クロスプラットフォーム対応のファイルシステムウォッチャー |
| 永続化 (設定・ブックマーク) | **SQLite** (rusqlite) | 単一ファイル管理、軽量 |
| UI言語 | **i18next + react-i18next** | 日英切り替え対応 |

### 2.1 配布形態

- **zip配布** — 展開後すぐに実行可能、インストーラー不要
- `.exe` + 同梱DLL/リソースをzipにパッケージ
- 設定・インデックスデータはアプリ実行ディレクトリ配下の `data/` に保存（ポータブル運用）

### 2.2 TextMate文法ファイルのライセンス対応

VS Codeリポジトリ（microsoft/vscode）の `extensions/` 配下のTextMate文法ファイルは大半がMITライセンス。同梱にあたり以下の対応を行う:

- 同梱する各文法ファイルのライセンスを調査し、一覧を `THIRD_PARTY_LICENSES` ファイルとしてアプリに同梱
- MITライセンス等、再配布可能なもののみを同梱対象とする

## 3. 機能要件

### 3.1 全文検索機能（コア）

#### 3.1.1 インデックス管理

| 項目 | 仕様 |
|------|------|
| インデックス対象 | バイナリ以外のすべてのファイル（テキスト判定はファイル先頭バイト列で自動判定）。ファイルサイズ上限なし |
| 対応規模 | 数千〜数万ファイル（中規模プロジェクト） |
| 永続化 | ディスク保存（`data/indexes/<workspace_id>/`） |
| リアルタイム更新 | `notify`クレートによるファイルシステム監視。変更検知後にインクリメンタルインデックス更新 |
| 日本語対応 | Tantivyの`lindera`トークナイザでN-gramまたは形態素解析による分かち書き |
| エンコーディング | `chardetng`で自動判定 → `encoding_rs`でUTF-8に変換してからインデックス登録 |
| 除外パターン | `.gitignore`形式の除外ルール設定可能（デフォルト: `.git/`, `node_modules/`, バイナリファイル等） |

#### 3.1.2 検索機能

| 項目 | 仕様 |
|------|------|
| 検索方式 | 全文検索（Tantivyクエリ）、正規表現検索（Rustのregexクレート併用） |
| 検索オプション | 大文字小文字区別、単語単位マッチ、正規表現モード |
| ファイルフィルタ | 含めるファイルパターン / 除外ファイルパターン（glob形式） |
| 検索結果表示 | VS Code準拠 — ファイルごとにグループ化し、マッチ行をインラインプレビュー表示。マッチ部分をハイライト |
| 検索履歴 | 直近の検索クエリ100件を永続保存し、検索入力欄からアクセス可能 |

### 3.6 検索サイドバー

VS Codeの検索サイドバーと同一のUI/UX。VS Codeのサイドバー仕様に準じ、サイドバー自体の開閉に加え、内部の各サブセクションも個別に開閉可能とする。

| 項目 | 仕様 |
|------|------|
| 位置 | アクティビティバーから切り替え（VS Code準拠） |
| サイドバー開閉 | アクティビティバーのアイコンクリックでサイドバー全体を開閉 |

検索サイドバーは以下の3つのサブセクションで構成される。各サブセクションはVS Code同様にヘッダクリックで個別に開閉可能。

#### 3.6.1 SEARCHセクション（検索）

| 項目 | 仕様 |
|------|------|
| セクションヘッダ | 「SEARCH」（開閉可能） |
| 入力欄 | 検索クエリ、置換クエリ（読み取り専用のため置換実行は不可、表示のみ） |
| オプション | 大文字小文字 / 単語単位 / 正規表現トグル |
| フィルタ | 含めるファイル / 除外ファイル（折りたたみ可能） |
| 結果 | ファイルごとにグループ化。ファイルの折りたたみ可能。マッチ数バッジ表示 |
| 結果クリック | マッチ行クリックで対象ファイルをエディタタブとして開き、該当行にジャンプ（VS Code準拠） |

#### 3.6.2 BOOKMARKSセクション（ブックマーク管理）

| 項目 | 仕様 |
|------|------|
| セクションヘッダ | 「BOOKMARKS」（開閉可能） |
| グループ表示 | 色別にグループ化してリスト表示。各色グループは折りたたみ可能 |
| リスト項目 | ファイル名 + 行番号 + 該当行のプレビューテキスト |
| ジャンプ | リスト項目クリックで対象ファイルを開き、該当行にスクロール |
| 一括操作 | 色グループ単位での一括削除 |
| ソート | ファイル名順 / 登録順（トグル） |

#### 3.6.3 HIGHLIGHTSセクション（ハイライトワード管理）

| 項目 | 仕様 |
|------|------|
| セクションヘッダ | 「HIGHLIGHTS」（開閉可能） |
| リスト表示 | 登録されたハイライト文字列を一覧表示。各項目に：文字列テキスト、検索モード（`ignoreCase` 等）、前方移動ボタン（`←`）、後方移動ボタン（`→`） |
| ナビゲーション `→` | 開いているファイルの現在のカーソル位置から**前方（下方向）**にある次のマッチ箇所にジャンプ |
| ナビゲーション `←` | 開いているファイルの現在のカーソル位置から**後方（上方向）**にある前のマッチ箇所にジャンプ |
| ファイル末尾/先頭到達時 | 末尾に到達したら先頭にラップ、先頭に到達したら末尾にラップ |
| コンテキストメニュー | リスト項目の右クリックで「削除」（対象のハイライト解除）/「全て削除」（全ハイライト一括解除） |

### 3.8 ワークスペース管理

| 項目 | 仕様 |
|------|------|
| ワークスペース定義 | 1つのルートディレクトリ = 1ワークスペース |
| 切り替え | メニューまたはショートカットでワークスペースを切り替え |
| ワークスペースリスト | 最近開いたワークスペース一覧を表示 |
| 独立管理 | インデックス・ブックマーク・検索履歴はワークスペースごとに独立 |
| 複数インスタンス起動 | アプリの複数起動を許可。異なるワークスペースはもちろん、同一ワークスペースも別インスタンスで同時に開くことが可能 |

#### 3.8.1 複数インスタンス時の排他制御

同一ワークスペースを複数インスタンスで開いた場合の競合を防止するため、以下の方針で排他制御を行う。

| 対象 | 方針 |
|------|------|
| Tantivyインデックス | インデックスの書き込み（構築・更新）は先にロックを取得したインスタンスのみが実行。他インスタンスは読み取り専用でインデックスを利用。書き込みロックが取得できない場合はトースト通知で「別のインスタンスがインデックスを更新中です」と案内 |
| SQLite（ブックマーク・検索履歴） | SQLiteのWALモードで同時読み書きに対応。書き込み競合時はリトライ（busy_timeout設定） |
| ファイル監視（notify） | 各インスタンスが独立してファイル監視を実行。インデックス書き込み権を持つインスタンスのみがインデックスを更新 |
| 設定ファイル（settings.json） | 最後に書き込んだインスタンスの設定が優先（last-write-wins）。設定変更時に他インスタンスへの反映はアプリ再起動または手動リロードで対応 |

### 3.11 ウェルカムタブ

### 7.2 SQLiteスキーマ

#### bookmarks テーブル

```sql
CREATE TABLE bookmarks (
  id            INTEGER PRIMARY KEY AUTOINCREMENT,
  workspace_id  TEXT NOT NULL,
  file_path     TEXT NOT NULL,
  line_number   INTEGER NOT NULL,
  color_index   INTEGER NOT NULL CHECK(color_index BETWEEN 0 AND 14),
  preview_text  TEXT,
  created_at    DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE(workspace_id, file_path, line_number)
);
CREATE INDEX idx_bookmarks_workspace ON bookmarks(workspace_id);
CREATE INDEX idx_bookmarks_color ON bookmarks(workspace_id, color_index);
```

#### search_history テーブル

```sql
CREATE TABLE search_history (
  id             INTEGER PRIMARY KEY AUTOINCREMENT,
  workspace_id   TEXT NOT NULL,
  query          TEXT NOT NULL,
  is_regex       BOOLEAN DEFAULT 0,
  case_sensitive BOOLEAN DEFAULT 0,
  whole_word     BOOLEAN DEFAULT 0,
  include_glob   TEXT,
  exclude_glob   TEXT,
  result_count   INTEGER,
  searched_at    DATETIME DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_history_workspace ON search_history(workspace_id, searched_at DESC);
```

## 9. リスク & 対策

| リスク | 影響 | 対策 |
|--------|------|------|
| TextMate文法のフロントエンド解析パフォーマンス | 大規模ファイルの描画遅延 | WebWorkerでの非同期解析、仮想スクロールの実装 |
| Tantivyの日本語トークナイズ精度 | 検索漏れ・ノイズ | linderaのIPADIC辞書利用 + 文字N-gramのハイブリッド方式の検討 |
| WebView2未インストール環境（Win10） | アプリ起動不可 | 初回起動時にWebView2 Bootstrapperでの自動インストールまたはREADMEでの案内 |
| ファイル監視のイベント溢れ | インデックス更新の遅延 | デバウンス処理（500ms）、バッチ更新 |
| ブックマークの行番号ずれ | ファイル変更後にブックマークが不正な位置を指す | 差分検知による行番号自動調整（ベストエフォート） |
| 同一ワークスペースの複数インスタンス同時操作 | インデックス破損、ブックマークの競合 | Tantivyは書き込みロックで排他制御、SQLiteはWALモード + busy_timeout でリトライ（3.8.1参照） |
