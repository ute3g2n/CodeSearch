# 09. テスト設計

基本設計書セクション13に基づくテスト戦略とテストケースの定義。

---

## 1. テスト戦略

### 1.1 テストピラミッド

```
          ┌─────┐
          │ E2E │  Tauri + WebDriver（主要フロー3〜5本）
         ─┴─────┴─
        ┌─────────┐
        │結合テスト│  サービス + DB/Index の結合（Rust: #[tokio::test]）
       ─┴─────────┴─
      ┌─────────────┐
      │  単体テスト  │  各モジュール個別（Rust: #[test] / TS: vitest）
     ─┴─────────────┴─
```

### 1.2 カバレッジ目標

| レイヤー | カバレッジ目標 | 理由 |
|---------|-------------|------|
| Rust services/ | 80%以上 | ビジネスロジックの中核 |
| Rust indexer/ | 80%以上 | 検索精度に直結 |
| Rust storage/ | 70%以上 | CRUD操作の正確性 |
| Rust commands/ | 50%以上 | 薄いレイヤーだが主要パスは検証 |
| TS stores/ | 80%以上 | UIロジックの中核 |
| TS components/ | 60%以上 | 主要コンポーネントの振る舞い |

## 5. テスト環境

### 5.1 Rust側

```toml
# Cargo.toml [dev-dependencies]
tokio = { version = "1", features = ["full", "test-util"] }
tempfile = "3"        # テスト用一時ディレクトリ
assert_matches = "1"  # パターンマッチアサーション
```

### 5.2 フロントエンド側

```json
// package.json (devDependencies)
{
  "vitest": "^1.0",
  "@testing-library/react": "^14",
  "@testing-library/user-event": "^14",
  "jsdom": "^24",
  "@tauri-apps/api": "mock" // Tauriコマンドのモック
}
```

### 5.3 モック戦略

| 対象 | モック方法 |
|------|----------|
| Tauri invoke | `vi.mock('@tauri-apps/api/core')` でinvokeをモック |
| Tauri listen | `vi.mock('@tauri-apps/api/event')` でlistenをモック |
| ファイルシステム | `tempfile` クレートで一時ディレクトリを使用 |
| SQLite | インメモリDB（`:memory:`）を使用 |
| Tantivy | RAMDirectoryを使用 |
